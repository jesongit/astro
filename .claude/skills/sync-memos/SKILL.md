# 同步 Memos 到博客

从用户的 Memos 实例中拉取内容，分析后制定整理计划，经用户确认后生成或更新博客文章。

## 前置要求

使用前需要用户提供以下信息，**不要使用硬编码值**，每次都需确认：

| 参数 | 说明 | 示例 |
|------|------|------|
| **URL** | Memos 实例地址 | `https://memos.example.com` |
| **Token** | API 访问令牌 | `eyJhbG...` |
| **时间范围** | 要拉取的 memo 时间段 | `最近一周`、`2024-01-01 到 2024-06-30`、`全部` |

如果用户没有提供以上任一参数，**必须主动询问**，不要猜测或使用默认值。

### 时间范围转换

用户可能使用自然语言描述时间范围，需要转换为 ISO 8601 格式：

| 用户说法 | 转换方式 |
|---------|---------|
| `最近一周` | 当前日期往前推 7 天 |
| `最近一个月` | 当前日期往前推 30 天 |
| `上个月` | 上月 1 日 00:00:00Z 到上月最后一天 23:59:59Z |
| `2024 年` | `2024-01-01T00:00:00Z` 到 `2024-12-31T23:59:59Z` |
| `全部` | 不传 `filter` 参数 |

## Memos API 说明

### 接口信息

- **请求方式**: `GET /api/v1/memos`
- **认证方式**: `Authorization: Bearer <TOKEN>`
- **分页参数**: `pageSize=50`，使用响应中的 `pageToken` 获取下一页
- **时间过滤**: 使用 `filter` 参数，格式为 CEL 表达式

### 时间过滤示例

```bash
# 获取指定日期之后的 memo
filter=create_time > "2024-01-01T00:00:00Z"

# 获取指定时间范围内的 memo
filter=create_time > "2024-01-01T00:00:00Z" && create_time < "2024-07-01T00:00:00Z"
```

### 请求示例

```bash
# 基础请求
curl -s -H "Authorization: Bearer <TOKEN>" \
  "<URL>/api/v1/memos?pageSize=50&filter=create_time+>+\"2024-01-01T00:00:00Z\""

# 翻页请求
curl -s -H "Authorization: Bearer <TOKEN>" \
  "<URL>/api/v1/memos?pageSize=50&pageToken=<NEXT_TOKEN>"
```

### 响应格式

```json
{
  "memos": [
    {
      "name": "memos/1",
      "content": "memo 的 Markdown 内容",
      "createTime": "2024-01-01T12:00:00Z",
      "updateTime": "2024-01-02T08:00:00Z",
      "resources": [
        {
          "name": "resources/1",
          "filename": "image.png",
          "externalLink": "https://...",
          "type": "image/png",
          "size": 12345
        }
      ]
    }
  ],
  "nextPageToken": "下一页 token，为空表示没有更多数据"
}
```

### 错误处理

| 状态码 | 含义 | 处理方式 |
|--------|------|---------|
| 401 | Token 无效或过期 | 提示用户重新提供 Token |
| 403 | 无权限 | 提示用户检查 Token 权限 |
| 404 | URL 错误 | 提示用户检查 Memos 实例地址 |
| 网络错误 | 无法连接 | 提示用户检查 URL 是否可访问 |

## 工作流程（分两阶段执行）

> **重要**：本 skill 分为【分析计划】和【执行写作】两个阶段。
> 第一阶段结束后必须停下来，等用户确认计划后才能进入第二阶段。

---

## 第一阶段：分析与制定计划

### 步骤 1：准备工作

1. **阅读 `BLOG_STYLE_GUIDE.md`** — 博客写作规范
2. **阅读 `src/data/blog/_drafts/memos-misc.md`** — 杂项草稿（检查已有内容，避免重复）
3. **扫描现有文章目录** — 执行 `find src/data/blog -name "*.md" | sort`，读取每篇文章的 frontmatter（`title`、`tags`、`description`），建立主题索引

### 步骤 2：拉取并读取所有 Memo

1. 使用用户提供的 URL 和 Token，按指定时间范围调用 API
2. 如有分页则自动翻页，拉取该时间段内所有 memo
3. 记录每条 memo：内容摘要、创建时间、是否含附件

### 步骤 3：逐条分析，确定处理方式

对每条 memo，按以下优先级判断归属：

#### 优先级顺序

| 优先级 | 处理方式 | 触发条件 |
|--------|---------|---------|
| 1 | **合并到已有文章** | memo 内容明确属于某篇已有文章的主题 |
| 2 | **重构已有文章** | 现有文章结构混乱或覆盖不完整，加入新内容后有必要重新整理 |
| 3 | **新建文章** | 多条相关 memo 可合并成独立主题，或单条内容足够充实 |
| 4 | **归入杂项草稿** | 零散、单薄、无法归类的内容 |
| 5 | **跳过** | 已在博客或杂项中存在的内容 |

#### 判断"合并"的依据

- memo 中的技术关键词与已有文章的 `tags` 或 `title` 匹配
- memo 内容主题与已有文章的某个章节相关
- 合并后不会破坏已有文章的结构和连贯性

#### 判断"重构"的依据

- 已有文章结构较松散，加入新内容后有必要重新组织章节
- 已有文章覆盖的主题范围变得更宽，需要调整标题或分类
- 已有文章的某些章节内容单薄，结合新 memo 内容重新撰写更合适

#### 内容质量评估（必做）

对每条将被处理的 memo，评估内容丰富程度：

- **充实**：内容完整，有步骤、有说明、有上下文，可直接扩充
- **一般**：有核心内容，但缺少背景、验证步骤或常见问题
- **精简**：仅有命令/代码片段或一两句话，需要大量补充

**评估结果决定是否需要联网资料补充**（见第二阶段步骤）。

### 步骤 4：输出计划，等待用户确认

分析完毕后，使用 `EnterPlanMode` 工具进入计划模式，用以下格式展示计划：

---

**共拉取 N 条 memo，有效内容 M 条，计划如下：**

##### 合并到已有文章

| Memo 摘要 | 目标文章 | 新增内容描述 | 内容质量 | 需联网补充 |
|-----------|---------|------------|---------|-----------|
| xxx | `notes/git-cheatsheet.md` | 新增「变基操作」章节 | 一般 | 是 |

##### 重构已有文章

| 目标文章 | 重构原因 | 重构方案简述 | 需联网补充 |
|---------|---------|------------|-----------|
| `notes/debian-handbook.md` | 新增 3 条 Docker 内容，原文结构需调整 | 拆分 Docker 章节，补充完整操作流程 | 是 |

##### 新建文章

| 文件名 | 标题 | 目录 | 来源 memo 数 | 内容质量 | 需联网补充 |
|--------|------|------|------------|---------|-----------|
| `redis-notes.md` | Redis 常用命令速查 | `notes/` | 3 | 一般 | 是 |

##### 归入杂项草稿

|-----------|------|

**等待用户确认后，再执行第二阶段。**

## 第二阶段：执行写作（用户确认后）

用户确认计划（或调整后确认）后，按计划逐项执行。

### 步骤 5：联网资料补充（按需）

对于计划中标注「需联网补充」的内容，在写作前先完成资料收集：

1. 使用 `WebSearch` 工具搜索相关资料
2. 搜索关键词示例：`"工具名 最佳实践"`、`"命令名 参数说明"`、`"错误信息 解决方案"`
3. 阅读搜索结果，提取有价值的补充内容：
   - 参数说明和最佳实践
   - 常见错误和解决方案
   - 官方文档链接
   - 版本兼容性说明
4. **不要照搬原文**，用自己的语言整理，引用时附上来源链接

> 联网补充的目的是让文章对读者真正有用，而不只是对 memo 的简单复述。

### 步骤 6：执行合并

对于「合并到已有文章」的内容：

1. 读取目标文章全文，理解结构和风格
2. 找到合适的章节插入，或新建一个章节
3. 整合 memo 内容 + 联网补充资料，使其匹配文章的风格和深度
4. 更新 frontmatter 中的 `modDatetime` 为当前日期（ISO 8601 格式）

### 步骤 7：执行重构

对于「重构已有文章」的内容：

1. 读取目标文章全文
2. 规划新的章节结构（更合理的组织方式）
3. 保留原有内容的核心信息，合并新 memo 内容
4. 按需联网补充，使各章节内容完整
5. 更新 frontmatter 中的 `modDatetime`
6. 若标题或 `description` 也需要更新，一并修改

### 步骤 8：新建文章

对于「新建文章」的内容：

#### 1. 确定文件存放目录

| 目录 | 用途 |
|------|------|
| `tutorials/` | 部署教程、入门指南 |
| `notes/` | 学习笔记、速查手册 |
| `leetcode/` | LeetCode 题解 |
| `_drafts/` | 内容不足，设 `draft: true` |

#### 2. 编写 Frontmatter

```yaml
---
author: Posase
pubDatetime: 2026-04-16T00:00:00Z   # 使用执行时的当前日期
title: "文章标题"
draft: false
tags:
  - Tag1
  - Tag2
description: "50-160 字符的文章摘要"
---
```

#### 3. 撰写正文

- 融合 memo 原始内容 + 联网补充资料
- 遵循内容结构规范（教程类 / 笔记类，见下方）
- 每个概念解释清楚"为什么"，不只是"怎么做"

**教程类文章结构**：

```
## 前言/背景          ← 为什么写这篇，读者能得到什么
## 前置条件           ← 需要什么环境/工具/知识
## 安装/配置          ← 分步骤操作，每步有代码和说明
## 验证              ← 如何确认操作成功
## 常见问题          ← 可能踩的坑和解决方案
## 参考              ← 官方文档和引用来源
```

**笔记类文章结构**：

```
## 安装/准备          ← 环境搭建（如有）
## 主题章节 1-N       ← 按主题分章节，不要一锅粥
## 其他/补充          ← 零散但有用的内容
```

### 步骤 9：更新杂项草稿

对于剩余零散内容：

1. 追加到 `src/data/blog/_drafts/memos-misc.md`
2. 按 `##` 标题分类组织
3. 不要重复已有的内容
4. 更新 frontmatter 中的 `modDatetime`

### 步骤 10：验证并汇报

1. 运行 `npx astro check` 确认构建无误
2. 向用户展示变更摘要

## 内容质量要求

### 内容扩充原则

- **补充原因**：解释"为什么"，不只是"怎么做"
- **补充前置条件**：读者需要什么基础知识或环境？
- **补充验证步骤**：操作完成后如何确认成功？
- **补充常见问题**：可能遇到的坑和解决方案
- **补充参考链接**：官方文档、相关教程
- **有时效性的内容**：添加提醒 `> **注意：本文内容具有时效性，xxx 可能已变化。**`

### 中英文排版规范

- 中英文之间加空格：`使用 Docker 部署` ✅ / `使用Docker部署` ❌
- 中文与数字之间加空格：`共 10 个` ✅ / `共10个` ❌
- 英文与数字之间不加空格：`Docker 3.0` ✅
- 行内代码前后加空格：`` 使用 `docker` 命令 `` ✅
- 全角标点前后不加空格
- 中文语境使用全角标点：`，。！？：；「」`
- 代码/命令/英文语境使用半角标点

### Markdown 规范

- **标题**：从 `##`（h2）开始，不跳级（`## → ### → ####`）
- **代码块**：必须标注语言（`bash`、`yaml`、`json`、`python`、`go` 等）
- **列表**：有序列表用于步骤，无序列表用于并列项目
- **链接**：使用描述性文字 `[官方文档](https://...)` ✅

### SEO 发布前检查

- [ ] `title` 包含核心关键词，30-60 字符
- [ ] `description` 独立撰写，50-160 字符，不与标题重复
- [ ] `tags` 2-5 个，使用标准化标签
- [ ] 正文标题层级正确（h2 起步，不跳级）
- [ ] 代码块标注语言类型
- [ ] 中英文之间有空格
- [ ] 无拼写错误
- [ ] 外部链接可访问
- [ ] 有时效性的内容添加了提醒

## 示例：Memo 到文章的转换

### 原始 memo（精简型）

```
docker run -d --name xxx -p 8080:80 nginx
记得加 --restart=always
```

### 联网搜索后扩充的文章内容

```markdown
## 部署 Nginx 容器

使用 Docker 快速部署一个 Nginx 服务：

\`\`\`bash
docker run -d \
  --name nginx-server \
  --restart=always \    # 宿主机重启后自动恢复容器
  -p 8080:80 \          # 映射端口：宿主机 8080 → 容器 80
  -v ./nginx/conf:/etc/nginx/conf.d \     # 挂载配置目录
  -v ./nginx/html:/usr/share/nginx/html \ # 挂载静态文件
  nginx:latest
\`\`\`

> **最佳实践**：生产环境建议始终添加 `--restart=always`，
> 确保服务在宿主机重启后自动恢复。挂载配置和静态文件目录
> 到宿主机，方便后续维护，无需进入容器操作。

### 验证部署

\`\`\`bash
# 检查容器状态
docker ps | grep nginx-server

# 测试访问
curl -I http://localhost:8080
\`\`\`

### 常见问题

**端口被占用**：将 `-p 8080:80` 改为其他未被占用的端口，如 `-p 8081:80`。
```

### 转换要点

| 原始 memo | 扩充后 |
|-----------|--------|
| 只有一行命令 | 添加了完整参数和注释 |
| 没有说明用途 | 补充了章节标题和说明文字 |
| 缺少最佳实践 | 添加了挂载卷、Tips 提示 |
| 没有解释参数 | 每个参数都有行内注释 |
| 没有验证步骤 | 补充了验证命令 |
| 没有常见问题 | 补充了端口冲突解决方案 |
