# 同步 Memos 到博客

从用户的 Memos 实例中拉取内容，整理成博客文章或归入杂项草稿。

## 前置要求

使用前需要用户提供以下信息，**不要使用硬编码值**，每次都需确认：

| 参数 | 说明 | 示例 |
|------|------|------|
| **URL** | Memos 实例地址 | `https://memos.example.com` |
| **Token** | API 访问令牌 | `eyJhbG...` |
| **时间范围** | 要拉取的 memo 时间段 | `最近一周`、`2024-01-01 到 2024-06-30`、`全部` |

如果用户没有提供以上任一参数，**必须主动询问**，不要猜测或使用默认值。

### 时间范围转换

用户可能使用自然语言描述时间范围，需要转换为 ISO 8601 格式：

| 用户说法 | 转换方式 |
|---------|---------|
| `最近一周` | 当前日期往前推 7 天 |
| `最近一个月` | 当前日期往前推 30 天 |
| `上个月` | 上月 1 日 00:00:00Z 到上月最后一天 23:59:59Z |
| `2024 年` | `2024-01-01T00:00:00Z` 到 `2024-12-31T23:59:59Z` |
| `全部` | 不传 `filter` 参数 |

## Memos API 说明

### 接口信息

- **请求方式**: `GET /api/v1/memos`
- **认证方式**: `Authorization: Bearer <TOKEN>`
- **分页参数**: `pageSize=50`，使用响应中的 `pageToken` 获取下一页
- **时间过滤**: 使用 `filter` 参数，格式为 CEL 表达式

### 时间过滤示例

```bash
# 获取指定日期之后的 memo
filter=create_time > "2024-01-01T00:00:00Z"

# 获取指定时间范围内的 memo
filter=create_time > "2024-01-01T00:00:00Z" && create_time < "2024-07-01T00:00:00Z"
```

### 请求示例

```bash
# 基础请求
curl -s -H "Authorization: Bearer <TOKEN>" \
  "<URL>/api/v1/memos?pageSize=50&filter=create_time+>+\"2024-01-01T00:00:00Z\""

# 翻页请求
curl -s -H "Authorization: Bearer <TOKEN>" \
  "<URL>/api/v1/memos?pageSize=50&pageToken=<NEXT_TOKEN>"
```

### 响应格式

```json
{
  "memos": [
    {
      "name": "memos/1",
      "content": "memo 的 Markdown 内容",
      "createTime": "2024-01-01T12:00:00Z",
      "updateTime": "2024-01-02T08:00:00Z",
      "resources": [
        {
          "name": "resources/1",
          "filename": "image.png",
          "externalLink": "https://...",
          "type": "image/png",
          "size": 12345
        }
      ]
    }
  ],
  "nextPageToken": "下一页 token，为空表示没有更多数据"
}
```

### 错误处理

| 状态码 | 含义 | 处理方式 |
|--------|------|---------|
| 401 | Token 无效或过期 | 提示用户重新提供 Token |
| 403 | 无权限 | 提示用户检查 Token 权限 |
| 404 | URL 错误 | 提示用户检查 Memos 实例地址 |
| 网络错误 | 无法连接 | 提示用户检查 URL 是否可访问 |

## 工作流程

### 第一步：准备工作

在开始之前，必须先完成以下准备：

1. **阅读 `BLOG_STYLE_GUIDE.md`** — 博客写作规范（标题、标签、排版、结构等全部规则）
2. **阅读 `src/data/blog/_drafts/memos-misc.md`** — 杂项草稿文件（检查已有内容，避免重复）
3. **扫描现有文章目录** — 了解博客中已有哪些文章，用于判断 memo 内容是否可以合并

当前博客文章目录结构：

```
src/data/blog/
├── tutorials/    ← 部署教程、入门指南
├── notes/        ← 学习笔记、速查手册
├── leetcode/     ← LeetCode 题解
└── _drafts/      ← 草稿（不会发布）
```

执行 `find src/data/blog -name "*.md" | sort` 获取完整文件列表，阅读每篇文章的 frontmatter（`title` 和 `tags`）以了解覆盖的主题范围。

### 第二步：拉取并分析 Memo

1. 使用用户提供的 URL 和 Token，按指定时间范围调用 API
2. 如有分页则自动翻页，拉取该时间段内的所有 memo
3. 阅读每条 memo 的 `content` 字段（Markdown 格式）
4. **去重检查**：将 memo 内容与以下内容对比，跳过已存在的：
   - `_drafts/memos-misc.md` 中的杂项内容
   - 已有文章中的相关章节（通过关键词匹配判断）
5. **处理附件**：如果 memo 包含 `resources`（图片/文件），记录但暂不处理，在汇报中提醒用户手动处理
6. 对每条新 memo 进行分类（见下方分类规则）

### 分类规则与优先级

按以下优先级判断每条 memo 的归属：

1. **优先合并到已有文章** — 如果 memo 内容明确属于某篇已有文章的主题（如 Docker 技巧属于 `debian-handbook.md`，Git 命令属于 `git-cheatsheet.md`），优先合并
2. **多条相关 memo 合并成新文章** — 如果多条 memo 围绕同一个主题（如都是关于 Redis 的），应合并成一篇新文章，而不是每条单独处理
3. **单条 memo 独立成文** — 仅当内容足够充实且聚焦时
4. **归入杂项** — 以上都不适用的零散内容

**判断"可合并"的依据**：
- memo 中的技术关键词与已有文章的 `tags` 或 `title` 匹配
- memo 的内容主题与已有文章的某个章节相关
- 合并后不会破坏已有文章的结构和连贯性

### 第三步：检查杂项草稿文件

检查 `_drafts/memos-misc.md` 中是否有已有内容可以与新 memo 合并，或者归入已有/新建文章：

- 如有，从 `memos-misc.md` 中提取并合并到目标文章
- 删除已提取的内容，避免重复

### 第四步：合并到已有文章

对于属于已有文章的内容：

1. 阅读目标文章全文，了解结构和风格
2. 找到合适的章节插入，或新建一个章节
3. 调整 memo 内容，使其匹配文章的风格和深度
4. **对内容进行扩充和润色**（详见下方「内容质量要求」）
5. 更新 frontmatter 中的 `modDatetime` 为当前日期（ISO 8601 格式，如 `2026-04-12T00:00:00Z`）

### 第五步：新建文章

对于内容充实、值得独立成文的 memo（或多条相关 memo 合并）：

#### 1. 确定文件存放目录

| 目录 | 用途 | 示例 |
|------|------|------|
| `tutorials/` | 部署教程、入门指南 | `docker-hysteria2.md` |
| `notes/` | 学习笔记、速查手册 | `golang-notes.md` |
| `leetcode/` | LeetCode 题解 | `leetcode-273.md` |
| `_drafts/` | 内容不足，设 `draft: true` | `electron-notes.md` |

#### 2. 编写 Frontmatter

```yaml
---
author: Posase
pubDatetime: 2026-04-12T00:00:00Z   # 使用执行时的当前日期，ISO 8601 格式
title: "文章标题"                     # 30-60 字符，简洁明确
draft: false                          # 内容不足时设为 true
tags:
  - Tag1                              # 2-5 个标签，使用标准化标签
  - Tag2
description: "50-160 字符的文章摘要"   # 不要和标题重复
---
```

**Frontmatter 规则**：
- `title`：30-60 字符，不使用 `「」` 前缀，直接表达内容
- `description`：50-160 字符，不要和标题重复，要能回答"这篇文章讲什么"
- `tags`：2-5 个，从 `BLOG_STYLE_GUIDE.md` 的标准化标签列表中选择
- `pubDatetime`：使用执行时的当前日期（不是 memo 的创建时间，因为文章是新整理的）
- `modDatetime`：新建文章不需要此字段，仅在后续更新时添加

#### 3. 标题命名规则

| 文章类型 | 标题风格 | 示例 |
|---------|---------|------|
| 部署教程 | `工具名 + 动作 + 目标` | Docker 部署 Hysteria2 代理服务 |
| 入门指南 | `工具名 + 入门/上手/指南` | Neovim + LazyVim 快速上手 |
| 学习笔记 | `语言名 + 学习笔记` | Golang 学习笔记 |
| 速查手册 | `工具名 + 速查/手册/详解` | Git 常用命令速查 |

#### 4. 文件名规则

使用 kebab-case 英文：`主题-关键词.md`（如 `docker-compose-guide.md`）

#### 5. 内容结构

**教程类文章**：
```
## 前言/背景          ← 为什么写这篇，读者能得到什么
## 前置条件           ← 需要什么环境/工具/知识
## 安装/配置          ← 分步骤操作，每步有代码和说明
## 验证              ← 如何确认操作成功
## 常见问题          ← 可能踩的坑和解决方案
## 参考              ← 官方文档和引用来源
```

**笔记类文章**：
```
## 安装/准备          ← 环境搭建（如有）
## 主题章节 1-N       ← 按主题分章节，不要一锅粥
## 其他/补充          ← 零散但有用的内容
```

#### 6. 内容不足的处理

- 放到 `_drafts/` 目录，设置 `draft: true`
- 文件开头添加 `<!-- [建议补充] 原因说明 -->`

### 第六步：更新杂项草稿

对于剩余的零散内容：

1. 追加到 `src/data/blog/_drafts/memos-misc.md`
2. 按 `##` 标题分类组织
3. 不要重复已有的内容
4. 更新 frontmatter 中的 `modDatetime`

### 第七步：验证并汇报

1. 运行 `npx astro check` 确认构建无误
2. 向用户展示变更摘要（使用表格）：

**新建文章**：

| 文件 | 标题 | 目录 | 来源 memo 数 |
|------|------|------|-------------|
| `xxx.md` | xxx | `notes/` | 3 |

**更新文章**：

| 文件 | 新增内容 |
|------|---------|
| `xxx.md` | 新增 xxx 章节 |

**杂项草稿**：
- 新增了 xxx 条内容

**跳过的 memo**：

| memo 摘要 | 跳过原因 |
|-----------|---------|
| xxx | 已存在于 xxx.md |

**含附件的 memo**（需用户手动处理）：

| memo 摘要 | 附件 |
|-----------|------|
| xxx | image.png (123KB) |

3. **不要自动提交或推送** — 等用户确认后再执行（项目规则）

## 内容质量要求

把 memo 扩充为文章时，必须遵守以下规则：

### 内容扩充

- **补充原因**：解释"为什么"，不只是"怎么做"。Memo 通常只记录操作步骤
- **补充前置条件**：读者需要什么基础知识或环境？
- **补充验证步骤**：操作完成后如何确认成功？
- **补充常见问题**：可能遇到的坑和解决方案
- **补充参考链接**：官方文档、相关教程等
- **有时效性的内容**：添加提醒 `> **注意：本文内容具有时效性，xxx 可能已变化。**`

### 中英文排版规范

- 中英文之间加空格：`使用 Docker 部署` ✅ / `使用Docker部署` ❌
- 中文与数字之间加空格：`共 10 个` ✅ / `共10个` ❌
- 英文与数字之间不加空格：`Docker 3.0` ✅
- 行内代码前后加空格：`` 使用 `docker` 命令 `` ✅
- 全角标点前后不加空格
- 中文语境使用全角标点：`，。！？：；「」`
- 代码/命令/英文语境使用半角标点
- 不要中英文标点混用

### Markdown 规范

- **标题**：从 `##`（h2）开始，不跳级（`## → ### → ####`）
- **代码块**：必须标注语言（`bash`、`yaml`、`json`、`python`、`go`、`erlang` 等），非显而易见的行要加注释
- **列表**：有序列表用于有步骤顺序的内容，无序列表用于并列项目
- **链接**：使用描述性文字 `[官方文档](https://...)` ✅ / `[点我](https://...)` ❌
- **行内代码**：用反引号包裹命令、文件名、变量名、端口号；不要包裹普通概念词汇

### SEO 发布前检查

每篇新建/修改的文章，完成后对照检查：

- [ ] `title` 包含核心关键词，30-60 字符
- [ ] `description` 独立撰写，50-160 字符，不与标题重复
- [ ] `tags` 2-5 个，使用标准化标签
- [ ] 正文标题层级正确（h2 起步，不跳级）
- [ ] 代码块标注语言类型
- [ ] 中英文之间有空格
- [ ] 无拼写错误
- [ ] 外部链接可访问
- [ ] 有时效性的内容添加了提醒

## 示例：Memo 到文章的转换

### 原始 memo

```
docker run -d --name xxx -p 8080:80 nginx
记得加 --restart=always
```

### 扩充后的文章内容

```markdown
## 部署 Nginx 容器

使用 Docker 快速部署一个 Nginx 服务：

\`\`\`bash
docker run -d \
  --name nginx-server \
  --restart=always \    # 宿主机重启后自动恢复容器
  -p 8080:80 \          # 映射端口：宿主机 8080 → 容器 80
  -v ./nginx/conf:/etc/nginx/conf.d \   # 挂载配置目录
  -v ./nginx/html:/usr/share/nginx/html \ # 挂载静态文件
  nginx:latest
\`\`\`

> Tips: 生产环境建议始终添加 `--restart=always`，确保服务在宿主机重启后自动恢复。
> 挂载配置和静态文件目录到宿主机，方便后续维护和更新，无需进入容器操作。
```

### 转换要点

| 原始 memo | 扩充后 |
|-----------|--------|
| 只有一行命令 | 添加了完整参数和注释 |
| 没有说明用途 | 补充了章节标题和说明文字 |
| 缺少最佳实践 | 添加了挂载卷、Tips 提示 |
| 没有解释参数 | 每个参数都有行内注释 |
